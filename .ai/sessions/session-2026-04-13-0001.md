### Design sketch 4 문서 요약

#### 01_core_abstractions.md (530 L)

**6개 core protocol** 정의:

1. **`SubAgentRole`** — 역할은 **데이터**(prompt, tool_allowlist, model_tier, output_schema, max_iterations). 실행은 Orchestrator가 함
2. **`ToolAdapter`** — 내부 구현(shell)과 외부 프록시(MCP client) 둘 다 동일 protocol. `async call → ToolResult(ok, value, error, duration_ms, metadata)`
3. **`Orchestrator`** — `run_pipeline`(static DAG) + `run_loop`(dynamic) + `invoke_role`(원자). 두 모드가 같은 invoke_role을 composition
4. **`ModelRouter`** — tier가 string, application 자유 정의. `SingleModelRouter` / `TieredModelRouter` 둘 다 기본 제공
5. **`MemoryStore`** — tier 이름 configurable, 3-tier 강제 X. SQLite + FTS5 기본 구현. `NullMemoryStore` 제공
6. **`HITLChannel`** — `ask`(blocking async) + `notify`(non-blocking push). Terminal / A2A SSE / Null 채널 구현

**핵심 결정 3가지**:
- 역할 = 데이터, 실행 = Orchestrator (OOP 상속 회피)
- Static pipeline의 edge는 Python 함수 (YAML DSL 만들지 않음)
- Dynamic mode의 driver role = `delegate` tool을 가진 일반 role

#### 02_coding_agent_mapping.md (551 L)

원본 `coding_agent/` 67개 파일을 6 protocol에 매핑. **library 35 / application 27 / split 5**.

**False-neutral 6가지 발견**:
1. `memory/store.py`의 `layer`/`project_id` → `tier`/`scope`
2. `progress_guard`의 TASK-NN regex → `key_extractor: Callable` 주입
3. `factory.py`의 `_FORK_RULES` (output contract) → application
4. `models.py`의 4-tier 리터럴 → tier 이름은 application
5. `memory/extractor.py`의 coding-specific 프롬프트 → application hook
6. `orchestrator.py`의 direct-vs-delegate 분류 → application entry point

특히 `memory/store.py`의 schema에 `project_id` 컬럼이 박혀 있는 게 가장 명백한 false-neutrality였음. apt-legal에서 "단지 ID"를 project_id에 넣는 건 의미 오염.

#### 03_apt_legal_mapping.md (682 L)

apt-legal-agent를 minyoung-mah 위에 **처음부터** 설계.

**보존 요구사항**: 분쟁 유형 10종, A2A endpoint, MCP 6 tools, gpt-4o, 개인정보 비저장, 면책 문구, k8s 배포
**재설계 가능**: 역할 분해, 키워드 폴백, 호출 순서, 에러 처리

**구조**:
- 3개 SubAgentRole: classifier (structured) → retrieval_planner (structured) → responder (structured)
- 1개 ExecuteToolsStep (LLM 없음, MCP tool fan-out)
- 6개 McpProxyToolAdapter (search_law, get_law_article, ...)
- StaticPipeline 선언만 application
- A2A는 Orchestrator 위에 얹히는 application layer

**얻는 것**: 코드 ~2500줄 → **~800줄** (library가 orchestration/retry/timeout/parallel/state machine 흡수)

3 데모 시나리오(층간소음 단순 / 복합 / 재건축)에 대해 pipeline 실행 추적까지 포함.

#### 04_open_questions.md (437 L)

매핑 과정에서 나온 **22개 결정 대기 항목**을 9개 주제로 정리. 각 항목에 옵션, tradeoff, **권장안** 명시.

가장 중요한 결정 2가지:

- **J1. 구현 순서**: library 뼈대 → apt-legal greenfield → coding agent 이식. 이유: apt-legal이 첫 실사용자가 되어 API gap을 발견한 후 coding 이식이 자연스러움. coding은 이미 잘 돌고 있음.
- **J2. Repo 구조**: minyoung-mah/examples/ 안에서 시작, Phase 4 이후 별도 repo 분리. 이유: Phase 3 동안 API가 유동적이므로 atomic commit 필요.

**기술적 결정 권장안**:
- A4: `output_schema + max_iterations=1` → fast path 분리 (`with_structured_output`)
- C1: `ToolResult.value: str | BaseModel | dict`
- C2: retry는 tool-level(transient) + role-level(semantic) 둘 다
- D1: memory schema migration → 기존 DB 버림 (clean start)
- D2: `memory_extractor: ... | None` 허용
- F2: watchdog timeout은 role-level dict (`role_timeouts={"classifier": 10, "coder": 180, ...}`)
- G1: observer event schema 표준화 (이름 + 공통 필드, backend adapter)

전체 22개 항목 표는 04 문서 §결정 요약 표 참조.
