## 사용자와 정립된 결정 사항 (이번 세션)

1. **minyoung-mah는 순수 library** — 소비자 애플리케이션 코드는 이 리포에 들어오지 않는다. 이 원칙은 앞으로 모든 의사결정의 최우선 기준.
2. **소비자는 별도 리포 + editable install** — `ax-coding-agent`, `apt-legal-agent`는 각자 자기 리포에서 `pip install -e ../minyoung-mah`로 소비. library에서 부족한 게 있으면 소비자 리포에서 이슈/PR로 피드백해 library에 흡수 (co-design은 리포 경계를 넘지 않음).
3. **기존 데모 코드는 삭제 대신 archive 보존** — git history와 참조 가치를 유지. 단, active 경로에 참여하지 않고 되살리지도 않음.
4. **실제 `apt-legal-agent`는 stale 설계와 분리** — 실제 소비자 리포(`../apt-legal-agent/`)는 Phase 2에 router/legal_lookup/domain_lookup/synthesizer 4-role + 2 MCP 서버 shape으로 새로 설계됨. minyoung-mah의 `docs/design/03_apt_legal_mapping.md`(구 설계)는 따르지 않음.
5. **`minyoung_mah/{resilience,memory}/__init__.py`의 broken re-export 제거는 유지** — library 자체 부채 해소. archive 이동과 무관한 독립 수리.
6. **`examples/coding_agent/` 이식(구 Phase 4)도 폐기** — 해당 작업은 `ax-coding-agent` 리포에서 minyoung-mah 소비로 전환하는 방식으로, 소비자 측에서 진행.

### Co-design 원칙의 재정의

이전 세션(2026-04-14)에 검증된 "한 리포에 library + application을 같이 두면 atomic commit으로 library 확장이 쉽다"는 주장은 이번 세션에 **번복**되었다. 이유: 단일 리포 co-design은 library가 특정 소비자의 shape에 이끌려 결합이 생기기 쉽고, 그 소비자가 stale해지면 library가 박제된 가정에 묶인다. 실제로 이번 세션에 그 현상을 관찰함.

새 co-design 원칙: library와 소비자는 **별도 리포 + PR 주기**로 진화한다. 소비자 리포에서 gap이 드러나면 minyoung-mah에 이슈로 올리고 library PR로 먼저 머지, 그 후 소비자가 새 library 버전을 당긴다. atomic commit은 희생하지만 경계가 유지된다.

단, 이 원칙이 강제하는 속도 저하가 부담스러워지면 재논의. 현재는 library가 protocol-level 안정화 단계라 소비자 의존이 많지 않아 이 모델로 충분.

### `ExecuteToolsStep`은 library에 남아 있다

지난 세션에 apt-legal 구현 중 발견된 gap으로 library에 추가된 `ExecuteToolsStep` + 5 tests는 `archive` 이동의 영향을 받지 않는다. 여전히 `minyoung_mah/core/` 하위에 있고 `tests/library/test_execute_tools_step.py`가 active. 경계 재정의 후에도 library 이득으로 남은 유일한 co-design 산출물.

## 이 세션에서 클로드가 가장 신경 쓴 것

**경계 존중**. 지난 세션에 만든 `examples/apt_legal_agent/`를 A2A로 확장하는 작업을 거의 다 끝낸 직후에 사용자가 "왜 library가 소비자를 신경쓰냐"는 근본적 질문을 던졌고, 그 순간 **방금 만든 코드에 대한 감정적 애착 대신 원칙을 우선**하는 결정을 내리는 것이 이번 세션의 핵심이었다. 삭제 대신 archive로 가는 타협안도 원칙과 매몰비용 사이의 균형으로, 사용자가 직접 선택한 지점. 이 경계 결정이 다음 많은 세션의 기준이 된다.

또한 **"library 수정은 최소로"** 원칙도 지켰다. 경계 재정의는 대부분 `git mv`와 문서 수정으로 끝났고, library 코드 자체는 (이미 수리된) `__init__.py` 2개를 제외하고 한 줄도 건드리지 않았다. archive 이동이 library 코드에 영향을 주지 않는다는 것이 증명된 것이 유일한 신뢰할 수 있는 회귀 안전망이었다.
