## 설계·경계 측면에서 신경 쓴 것

이번 세션의 목표는 **"코드에 박제된 암묵 규칙을 발굴해 문서로 이식"**이었다. 이미 세션 0001~0003을 통해 경계, LiteLLM 분할, 두 retry 레이어 같은 원칙이 4~5곳의 문서에 박제되어 있었지만, **디렉토리별 작업 가이드**와 **한 눈에 보는 아키텍처 투어**는 없었다. 그래서:

1. **각 서브모듈의 AGENTS.md는 해당 디렉토리 밖의 컨텍스트를 요구하지 않도록** 작성했다. 누군가 `minyoung_mah/observer/`만 보고 있어도 EVENT_NAMES 규칙, LiteLLM 분할, 4곳 동기화 규칙을 그 파일 안에서 알 수 있다.

2. **`docs/ARCHITECTURE.md`는 "이 라이브러리는 이런 것이다"와 "이 라이브러리는 이런 것이 아니다"를 같이 서술**했다. 경계 재정의(세션 0001)와 Phase 2b 클린업(세션 0002)에서 깨달은 것은, 긍정 정의만으론 경계가 다시 뭉개진다는 것. §9 "무엇이 라이브러리가 아닌가"를 의도적으로 길게 썼다.

3. **README의 문서 지도**는 "어느 문서를 언제 읽나"를 기준으로 정렬했다. "무엇을 담는지"로 정렬하면 독자가 자기 상황에 어떤 문서가 맞는지 매번 재판단해야 한다.

4. **코드 소스를 먼저 읽고 문서를 썼다.** `protocols.py` / `orchestrator.py` / `types.py` / `observer/events.py` / `resilience/policy.py` / `model/router.py` / `memory/store.py`(일부) / `hitl/channels.py`(일부)를 확인한 뒤 각 AGENTS.md와 ARCHITECTURE.md를 작성. 덕분에 ARCHITECTURE §3의 실행 경로 트리와 §5의 이벤트 표가 실제 코드와 일치한다.

## 이 세션에서 클로드가 가장 신경 쓴 것

**"문서는 코드 옆에 있을 때만 쓸모 있다."** 아키텍처 문서를 한 곳에 크게 쓰는 것과 각 디렉토리에 작은 `AGENTS.md`를 두는 것은 대체재가 아니라 보완재다. 큰 문서는 "전체를 이해하고 싶을 때", 작은 문서는 "이 디렉토리에서 한 줄 고치려 할 때". 둘 다 만들어야 한쪽이 stale되어도 다른 쪽이 최소한 방향을 잡아준다.

그리고 **README의 "문서 지도"**. 이건 단순 링크 모음이 아니라 "독자가 자기 상황에 맞는 문서를 재판단하지 않도록" 정렬했다. 미래의 세션에서 minyoung-mah가 처음 열릴 때, README → ARCHITECTURE → 필요한 AGENTS.md 순으로 자연스럽게 따라가도록 만든 것이 이번 세션의 장기 효용이다.
