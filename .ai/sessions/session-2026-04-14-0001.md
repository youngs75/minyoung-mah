### Library에 추가된 것 (co-design gap #1)

**`ExecuteToolsStep`** — LLM 없는 순수 tool 디스패치 pipeline step.

- `core/types.py`: `ExecuteToolsStep` dataclass, `PipelineStepResult.tool_results` 필드 추가, `StaticPipeline.steps: list[PipelineStep | ExecuteToolsStep]`
- `core/orchestrator.py`: `_run_step`이 isinstance로 분기, 새 `_run_execute_tools_step` 메서드가 priority grouping 실행
- 동작: `tool_calls_from(state) -> list[(ToolCallRequest, priority)]`을 받아 priority 오름차순 그룹으로 분할, 각 그룹은 `ToolInvocationEngine.call_parallel`로 병렬 실행, 그룹 간은 직렬
- `continue_on_failure=True` (기본): 개별 실패가 step을 abort하지 않음. responder가 결과에서 실패를 감지해 user에게 고지
- `continue_on_failure=False`: 첫 실패 priority 그룹에서 중단, `FAILED RoleInvocationResult` 합성해 pipeline abort 경로로 진입
- `depends_on`은 현재 priority 순서로만 해석. 런타임 args 재작성은 미구현 (다음 세션 후보)

### 테스트

- `tests/library/test_execute_tools_step.py` (5 tests)
  - parallel within priority (timing 검증, 80ms × 3 < 200ms)
  - priority groups run sequentially
  - continue_on_failure=True collects all results
  - continue_on_failure=False aborts pipeline, 후속 priority 그룹 실행 안 됨
  - unknown tool returns error result

- `examples/apt_legal_agent/tests/test_pipeline_e2e.py` (2 tests)
  - 시나리오 1: "층간소음 기준 데시벨" — 4 step 파이프라인 전체 구동, MCP 2 호출 순서 검증, `AgentResponse.disclaimer` 기본값 보존

### Co-design 원칙 작동 검증

Phase 3가 apt-legal을 먼저 시도했고 → 첫 번째 gap(`ExecuteToolsStep`)이 나타났고 → library에 atomic commit으로 추가됨. **이게 design doc 04 §J1에서 예측한 co-design 흐름**. 한 repo에 library + apt-legal이 같이 있어서 이 atomic commit이 가능했다는 점도 J2 결정의 정당성을 확인.

다음 세션의 A2A 작업에서 추가 library gap이 나올 가능성:
- `HITLChannel.ask`가 blocking async인 상태에서 SSE 이벤트 스트리밍을 어떻게 엮을지 (decision A3/E1 재검토 필요할 수 있음)
- Observer가 per-run trace ID를 발급하고 A2A task_id와 연결하는 방법
- `run_pipeline`이 중간 상태를 SSE로 push할 수 있게 하는 hook

이런 것들이 Phase 3 둘째 세션에서 "library에 addition 필요"로 떠오르면 즉시 반영한다.
