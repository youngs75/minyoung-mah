1. **제자리 클린업 > 새 리포** — 경계 재정의가 코드가 아니라 문서+구조 단위였기 때문에 git history/archive/origin의 서사를 보존하는 게 가치가 컸다. 새 리포는 깔끔함의 착시일 뿐.
2. **Deep Insight 개념은 "참고만, 구현은 소비자 요구 시"** — 선제적으로 `HierarchicalOrchestrator`, `QueueObserver`, `max_iterations` 같은 기능을 추가하지 않고, `05_reference_topologies.md`에 문서화만 해둠. 소비자 리포에서 동일 패턴이 **3회 이상 관찰될 때** library 흡수 고려.
3. **Langfuse는 LiteLLM 뒷단** — library가 Langfuse SDK를 직접 의존하지 않는다. LLM-level trace는 LiteLLM callback, orchestration-level trace는 library Observer. 직교 분할.

### Langfuse에 대한 확정된 원칙 (다음 세션 P0 작업의 근거)

- **library는 Langfuse SDK를 직접 import하지 않는다.** observer/events.py에는 `NullObserver`, `CollectingObserver`, `StructlogObserver`, `CompositeObserver` 4개만. 이게 "표준 벽지"다.
- **LLM-level trace는 LiteLLM의 자동 Langfuse 통합으로 커버된다.** 소비자가 `litellm.success_callback = ["langfuse"]`을 설정하면 prompt/completion/token usage가 자동 전송됨.
- **orchestration-level trace는 minyoung-mah의 Observer 프로토콜로 커버된다.** `orchestrator.run.*`, `role.invoke.*`, `pipeline.step.*`, `tool.call.*` 등의 canonical event name이 발행된다.
- 두 층은 **같은 trace_id**(LiteLLM이 request header에 넣거나, Observer metadata에 명시적으로 전달)로 Langfuse에서 연결될 수 있다. 이 구성은 **소비자 책임**이고 library는 그 훅 지점만 제공.
