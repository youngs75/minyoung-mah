## 4. 설계·경계 측면에서 신경 쓴 것

### 4.1. "library 답게"의 두 축

**소비자의 손을 덜어주는 API:**
- `shared_state`는 `complex_id` 같은 상수를 4번 복사하지 않게 함
- `payload_as[T]()`는 이중 속성 접근을 타입 안전하게 감춤
- `format_for_llm()`은 INCOMPLETE 배너 규칙을 **라이브러리 코드에 한 번만** 박제해서 모든 소비자가 같은 원칙으로 환각을 1차 방어하게 함

**거짓말 제거:**
- `ModelHandle = Any` → `ModelHandle = BaseChatModel` (이미 사용 중인 인터페이스를 타입에도 반영)
- optional `[langchain]` extra → required (어차피 소비자는 모두 설치할 수밖에 없었음)
- `run_loop` `NotImplementedError` → 삭제 (미래 구현 시 재설계 부채)

### 4.2. `format_for_llm`은 library 코드에 박제된 **규범**이다

시나리오 3 환각 주범은 synthesizer가 `INCOMPLETE` 단계의 부실한 output을 정상 결과처럼 받아먹은 것이었다. 프롬프트 수정이 2차 가드라면, library의 `format_for_llm`은 **1차 가드**다: 어떤 소비자가 작성한 synthesizer든, 이 헬퍼를 호출하면 upstream 역할의 status 배너가 자동으로 LLM에 전달된다. 두 번째 소비자가 같은 함정을 밟지 않도록 라이브러리가 미리 방어막을 친 것.

### 4.3. `shared_state` merge 방향 (pipeline-first-then-step)

`{**pipeline_shared_state, **ctx.shared_state}` — 즉 step이 override할 수 있다. 반대 방향(`{**ctx.shared_state, **pipeline_shared_state}`)도 가능하지만 그러면 step이 pipeline 상수를 덮어쓸 수 없어서 쓸모가 반감된다. apt-legal의 경우는 어차피 step별 override가 없어서 두 방향 모두 동작하지만, library로서는 **더 표현력 있는 방향**을 선택.

### 4.4. `run_loop` 삭제의 대안

동적 driver-role loop가 필요한 소비자는 `invoke_role`을 자기 `while` 루프로 감싸면 됩니다. library는 `ProgressGuard` + `ResiliencePolicy`만 제공해서 소비자가 조립할 수 있게 합니다. 이 방향이 더 정직한 이유: library가 loop shape를 미리 정하면 어떻게 해도 코딩 에이전트의 실제 루프와 맞지 않을 것이었음 (세션 0001에서 이미 예상한 패턴).

- `ExecuteToolsStep`의 실제 사용 사례 — apt-legal에서 router가 `ToolCallRequest` 플랜을 생성하면 그 뒤 tool 실행을 LLM 없이 병렬화할 수 있는지 검토
- `ProgressGuard` 기본 활성화 여부 재검토 (동적 loop 소비자가 생기면)

### 7.2. 0.1.0 breaking change 목록

소비자 관점에서 무엇이 깨지는가:

1. **`from minyoung_mah import LoopResult, LoopState`** → ImportError. 사용 소비자 없음 (apt-legal 0건).
2. **`orchestrator.run_loop(...)`** → AttributeError. 사용 소비자 없음.
3. **`ModelHandle = Any` 의존 코드** → 정적 타입 체커에서 타입 오류 가능. Runtime 동작 변화 없음.
4. **`default_resilience()`의 90s 기본값**에 암묵 의존하던 코드 → 새 기본값 180s. apt-legal은 어차피 override해서 쓰고 있어서 영향 없음.
5. **`langchain-core` 미설치 환경**에서 `minyoung_mah.core.protocols` import 불가. 사용 소비자 없음 (실행 시점엔 어차피 필요했음).

apt-legal 외의 소비자가 0개인 현재 시점에서는 모든 breaking change가 safe.

### 7.3. `0.1.0` 네이밍 결정

`1.0.0`은 너무 이름. `0.1.0`은 "첫 실소비자 피드백 반영"과 "소비자 2개가 쓰기 시작하기 전"이라는 현재 단계에 맞는 수준. 다음 소비자(또는 apt-legal Phase 3 기능)가 추가되면서 stabilize되면 0.2.x / 0.3.x로 올라갈 것.

### 7.4. atomic 2-리포 커밋 금지 재확인

본 세션에서 minyoung-mah 코드와 apt-legal 코드를 같은 시점에 수정했지만, **커밋은 리포 경계를 지킵니다**:
1. minyoung-mah 단일 커밋 → push → `v0.1.0` 태그 → 태그 push
2. apt-legal에서 `pyproject.toml`을 `v0.1.0`로 bump → 별도 커밋 → push

이 순서는 세션 0001·0003·0004에서 박제된 규칙의 직접 적용입니다.

### 7.5. 첫 소비자의 가치

apt-legal-agent가 없었다면 0.1.0의 3개 헬퍼는 **발견되지 않았을 것**입니다. 특히 `format_for_llm`의 INCOMPLETE 배너는 시나리오 3의 환각 디버깅(세션 0004) 없이는 library 작성자가 떠올리기 어려운 패턴이었습니다. **라이브러리는 사용되면서만 진화합니다**라는 원칙이 이번 세션에서 가장 진하게 확인됐습니다.

## 10. 이 세션에서 클로드가 가장 신경 쓴 것

**"내가 평가를 틀렸을 때 스스로 고치는 것."** 1차 평가에서 "consumer 검증 0건"이라고 단정한 것은 세션 0004 handoff 텍스트의 "별도 창에서 진행 중"이라는 문구만 근거로 삼은 판단이었다. 사용자가 `../apt-legal-agent`를 직접 확인해보라고 지시했을 때, 그걸 "이전 판단 방어"가 아닌 "데이터 추가"로 받아들여 평가를 전면 재작성한 것이 이번 세션의 방향을 바꿨다.

그리고 **"library를 library답게 만든다"는 요청을 구체적인 3개의 소비자 gap으로 치환한 것.** 추상적인 "더 library스럽게"는 끝없이 확장될 수 있지만, 소비자(apt-legal)에서 이미 드러난 gap 3건에 집중하면 **실측된 가치**를 낳는다. 이게 "premature abstraction 금지"와 "실사용이 라이브러리를 진화시킨다"가 동시에 만족되는 지점이었다.
