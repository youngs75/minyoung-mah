**즉 verifier가 같은 명령을 반복한 게 아니라 첫 호출이 default 300초를 다 쓴 것**. LLM은 timeout 마커를 보면 자체적으로 줄이는 학습 행동을 보임 → P0 90초 단축이 그 학습 트리거를 더 일찍 발화시킴.

## 핵심 교훈 (이번 세션의 결정적 통찰)

### 1. Harness 설계 철학 — 5가지 책임 + 형식 강제 폐기
사용자 정립:
> "주어진 역할에 맞게, 컨텍스트에 충실하게, LLM 스스로 알고있는 지식을 최대한 활용해 task를 정확하게 수행하라"

Harness 책임 5가지:
1. **안전 가드레일 (A형)** — 행동의 안전 경계만 강제 (execute watch 거부, fixer 도구 권한, 위험 경로)
2. **오동작 탐지 (circuit breaker)** — ProgressGuard task repeat, SubAgent 반복 호출, max_turns
3. **도구 입출력 명료성** — verifier exit code+stdout 그대로 노출 (요약 X, 노이즈도 X)
4. **컨텍스트 자동 전달** — `_user_decisions` prepend, B-1 자동 todo 마킹
5. **관찰 가능성** — Langfuse trace + structured agent.log

폐기된 패턴 (B형 형식 강제):
- SPEC 4섹션 강제, per-task GWT marker, 1200자 minimum, dod 25 checkbox, reference example 첨부

### 2. 자가 진단 6~10번 (memory에 보강)
Iteration 7 사례에서 24시간 안에 같은 함정에 빠진 것을 사용자가 외부 검증으로 잡음:

6. **"무엇을" vs "어떻게" 분리** — 가이드는 "무엇을 놓치지 마라"만, "어떻게 적용하라"는 LLM에 맡김
7. **명령형 단어 금지** — MUST/do not/always/never/interleave 같은 hard constraint 단어는 작은 강제
8. **키워드 list = 도메인 bias** — "TDD/BDD/SDD/CI-CD/microservices" 명시는 list 안의 것만 보게 만듦
9. **사용자가 안 쓴 단어 박지 말 것** — atomic/feature slice/module 같은 단어가 LLM 사고 기준이 됨
10. **5줄 룰** — 가이드 하나가 5줄을 넘으면 다시 의심

`feedback_harness_philosophy.md`에 자기 검증 사례까지 박음.

### 3. LLM의 자율 학습은 신호 강도에 비례
v8 trace에서 LLM이 first timeout 발화를 본 후 자체적으로 `timeout 60` prefix를 추가하는 학습 행동 관찰. **harness가 default를 줄이면 LLM 학습 트리거도 더 일찍 발화**. 즉 우리가 패턴 매칭으로 분기를 박는 것보다 정직한 신호(빠른 timeout)를 주는 게 더 효과적.

### 4. "갈수록 오래 걸린다"의 진짜 원인 — prompt 비대화 부채
SYSTEM_PROMPT 22줄 → 58줄, planner_prompt 13줄 → 53줄. 매 라운드 가이드를 추가하면서 슬림 약속을 망각. **약한 모델에서 input length는 latency에 거의 선형 비례**. 다음 라운드 P3 후보.
